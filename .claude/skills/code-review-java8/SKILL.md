# 코드 리뷰 표준 (Java 8 / Boot 2.x)

## 1. 심각도 분류

| 등급 | 기준 | 예 |
|------|------|-----|
| **Critical** | 데이터 훼손/보안 사고/장애 직결 | 소유권 검증 누락, 평문 비밀번호, 트랜잭션 밖 다단계 갱신 |
| **Major** | 버그 또는 운영 문제 가능성 높음 | null 미처리 분기, 잘못된 상태코드, ThreadLocal 미정리 |
| **Minor** | 동작하지만 표준 위반/유지보수 비용 | 네이밍, 중복, 레이어 책임 혼합 |
| **질문** | 확인 못 한 추측 — 단정 금지 | "이 경우 동시성 문제는 없나요?" |

## 2. 체크리스트

### 환경 제약 (이 조직 고정)
- [ ] Java 9+ 문법 없음: `var`, `record`, Text Block, `Stream.toList()`, switch expression
- [ ] `jakarta.*` import 없음 (`javax.*`만) / BOM 버전 덮어쓰기 없음
- [ ] `WebSecurityConfigurerAdapter` 등 deprecated API 미사용

### 정확성
- [ ] 경계값/null/빈 컬렉션 처리, 예외 흐름에서의 자원/상태 정리(finally)
- [ ] 트랜잭션 경계 적절성, RANDOM_PORT 테스트의 롤백 불가 인지
- [ ] equals 비교 방향(NPE), enum 이름 변경의 DB 영향

### 보안
- [ ] URL 인가 + 데이터 소유권 검증 양쪽 존재
- [ ] 응답/로그에 민감정보(비밀번호, 내부 예외 메시지) 노출 없음
- [ ] SQL 파라미터 바인딩(#{}) — ${} 문자열 치환 없음
- [ ] 상태 변경 엔드포인트의 CSRF 고려

### 테스트 품질
- [ ] 단언 없는 테스트(호출만 하고 검증 없음) 없음
- [ ] 예외 검증에 메시지/타입 모두 / "호출 안 됨"(never) 검증 누락 없음
- [ ] 테스트 간 실행 순서/데이터 의존 없음
- [ ] 순수 로직까지 Mock으로 덮은 과잉 Mocking 없음

## 3. 보고서 형식

```
## 리뷰 요약
(1~3문장 — 머지 가능/조건부/불가 의견 포함)

## 발견 사항
- [Critical] 파일:라인 — 내용. 권장: ...
- [Major] ...
- [질문] ...

## 좋은 패턴
- 파일:라인 — 팀 표준 승격 후보인 이유
```

원칙: 지적에는 반드시 "권장 수정 방향"을 붙인다. 확신 없으면 등급을 낮춘다 — 거짓 양성이 리뷰 신뢰를 죽인다.
