# 교육 문서 작성 표준 (범용)

## 1. 철학: 문제를 먼저 겪고, 해결책을 배운다

해결책(도구/패턴)부터 설명하지 않는다. 독자가 **문제를 느낀 뒤**에 도구를 소개한다.
각 문서/장의 시작은 "지금 방식의 한계" 또는 "실제로 겪는 곤란함"이다.

## 2. 표준 템플릿 (장 구성)

1. **Before We Start** — 왜 이게 필요한가: 현재 방식의 한계, 실제 문제 상황. 정확한 비유 1개(억지 비유 금지)
2. **What We're Building** — 이번에 만들/배울 것의 전체 그림 (디렉토리 트리, 표)
3. **Core Concepts** — 핵심 개념을 적은 수로 깊게. 코드와 함께. "왜"를 항상 동반
4. **Step-by-Step** — 실행 가능한 절차. 명령어와 예상 출력 포함. "일부러 깨뜨려보기" 실습 권장
5. **Lessons Learned** — 실제 겪은 버그/함정: `증상 → 원인 → 해결 → 교훈` + "시니어의 시선" 한 단락
6. **Key Takeaways** — 3~5개 불릿 요약
7. **Next Steps** — 다음 학습의 "문제"를 예고 (cliffhanger — 다음 단계의 동기 부여)

## 3. 작성 규칙

- 한국어 친근체, 기술 용어는 영어 유지. 독자를 "여러분"으로
- 모든 코드 블록: 실제 동작 확인된 것만 + 파일 경로 명시. 실행 출력은 실제 캡처 사용
- 표는 비교/선택 기준에만 — 흐름과 인과는 산문으로
- 분량 가이드: 문서당 학습 20~30분 (200~400줄). 넘치면 분할
- 문서 헤더에 소요 시간/선수 지식/코드 위치 명시
- 새 문서는 반드시 인덱스 문서(커리큘럼/README)에 등록

## 4. 금지 사항

- 검증 안 된 코드 예시, 매끈하게 성공한 척(실제 시행착오를 숨기는 것)
- 이론 나열만 있고 실습 없는 장
- 근거 없는 단정("항상", "절대") — 트레이드오프를 명시하라
