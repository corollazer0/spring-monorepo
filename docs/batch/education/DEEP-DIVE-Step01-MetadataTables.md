### 실제 DDL 스크립트

Spring Batch는 각 데이터베이스별로 **미리 작성된 DDL 파일**을 jar 안에 포함하고 있습니다.

**위치:** `spring-batch-core-4.3.x.jar` 내부
```
org/springframework/batch/core/
├── schema-h2.sql           (H2용)
├── schema-mysql.sql        (MySQL용)
├── schema-postgresql.sql   (PostgreSQL용)
├── schema-oracle10g.sql    (Oracle용)
└── ... (기타 DB별 스크립트)
```

**H2용 실제 DDL:**

```sql
-- ============================================================================
-- 1. BATCH_JOB_INSTANCE: Job의 고유 식별 정보
-- ============================================================================
CREATE TABLE BATCH_JOB_INSTANCE  (
	JOB_INSTANCE_ID BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
	VERSION BIGINT,
	JOB_NAME VARCHAR(100) NOT NULL,           -- Job 이름
	JOB_KEY VARCHAR(32) NOT NULL,             -- JobParameters 해시값
	CONSTRAINT JOB_INST_UN UNIQUE (JOB_NAME, JOB_KEY)  -- 중복 방지
);

-- 💡 왜 JOB_NAME + JOB_KEY를 UNIQUE로?
-- → 같은 Job을 같은 Parameters로 두 번 실행하는 것을 방지!

-- ============================================================================
-- 2. BATCH_JOB_EXECUTION: Job 실행 시도 기록
-- ============================================================================
CREATE TABLE BATCH_JOB_EXECUTION  (
	JOB_EXECUTION_ID BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
	VERSION BIGINT,
	JOB_INSTANCE_ID BIGINT NOT NULL,
	CREATE_TIME TIMESTAMP NOT NULL,           -- 생성 시간
	START_TIME TIMESTAMP DEFAULT NULL,        -- 시작 시간
	END_TIME TIMESTAMP DEFAULT NULL,          -- 종료 시간
	STATUS VARCHAR(10),                       -- COMPLETED, FAILED, STARTED 등
	EXIT_CODE VARCHAR(2500),                  -- 종료 코드
	EXIT_MESSAGE VARCHAR(2500),               -- 종료 메시지
	LAST_UPDATED TIMESTAMP,                   -- 마지막 업데이트
	JOB_CONFIGURATION_LOCATION VARCHAR(2500) NULL,
	CONSTRAINT JOB_INST_EXEC_FK FOREIGN KEY (JOB_INSTANCE_ID)
		REFERENCES BATCH_JOB_INSTANCE(JOB_INSTANCE_ID)
);

-- 💡 하나의 JobInstance는 여러 번 실행 시도 가능 (실패 후 재시도)

-- ============================================================================
-- 3. BATCH_JOB_EXECUTION_PARAMS: Job 실행 시 전달된 파라미터
-- ============================================================================
CREATE TABLE BATCH_JOB_EXECUTION_PARAMS  (
	JOB_EXECUTION_ID BIGINT NOT NULL,
	TYPE_CD VARCHAR(6) NOT NULL,              -- STRING, DATE, LONG, DOUBLE
	KEY_NAME VARCHAR(100) NOT NULL,           -- 파라미터 이름
	STRING_VAL VARCHAR(250),                  -- 문자열 값
	DATE_VAL TIMESTAMP DEFAULT NULL,          -- 날짜 값
	LONG_VAL BIGINT,                          -- Long 값
	DOUBLE_VAL DOUBLE PRECISION,              -- Double 값
	IDENTIFYING CHAR(1) NOT NULL,             -- 식별 파라미터 여부 (Y/N)
	CONSTRAINT JOB_EXEC_PARAMS_FK FOREIGN KEY (JOB_EXECUTION_ID)
		REFERENCES BATCH_JOB_EXECUTION(JOB_EXECUTION_ID)
);

-- 💡 하나의 Execution에 여러 파라미터가 들어갈 수 있음 (1:N 관계)

-- ============================================================================
-- 4. BATCH_STEP_EXECUTION: Step 실행 기록
-- ============================================================================
CREATE TABLE BATCH_STEP_EXECUTION  (
	STEP_EXECUTION_ID BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
	VERSION BIGINT NOT NULL,
	STEP_NAME VARCHAR(100) NOT NULL,          -- Step 이름
	JOB_EXECUTION_ID BIGINT NOT NULL,
	START_TIME TIMESTAMP NOT NULL,
	END_TIME TIMESTAMP DEFAULT NULL,
	STATUS VARCHAR(10),
	COMMIT_COUNT BIGINT,                      -- 커밋 횟수
	READ_COUNT BIGINT,                        -- 읽은 아이템 수
	FILTER_COUNT BIGINT,                      -- 필터링된 아이템 수
	WRITE_COUNT BIGINT,                       -- 쓴 아이템 수
	READ_SKIP_COUNT BIGINT,                   -- 읽기 Skip 수
	WRITE_SKIP_COUNT BIGINT,                  -- 쓰기 Skip 수
	PROCESS_SKIP_COUNT BIGINT,                -- 처리 Skip 수
	ROLLBACK_COUNT BIGINT,                    -- 롤백 횟수
	EXIT_CODE VARCHAR(2500),
	EXIT_MESSAGE VARCHAR(2500),
	LAST_UPDATED TIMESTAMP,
	CONSTRAINT JOB_EXEC_STEP_FK FOREIGN KEY (JOB_EXECUTION_ID)
		REFERENCES BATCH_JOB_EXECUTION(JOB_EXECUTION_ID)
);

-- 💡 성능 모니터링의 핵심! READ/WRITE/SKIP 카운트를 모두 기록

-- ============================================================================
-- 5. BATCH_STEP_EXECUTION_CONTEXT: Step ExecutionContext
-- ============================================================================
CREATE TABLE BATCH_STEP_EXECUTION_CONTEXT  (
	STEP_EXECUTION_ID BIGINT NOT NULL PRIMARY KEY,
	SHORT_CONTEXT VARCHAR(2500) NOT NULL,     -- 짧은 컨텍스트
	SERIALIZED_CONTEXT LONGVARCHAR,           -- JSON 직렬화된 데이터
	CONSTRAINT STEP_EXEC_CTX_FK FOREIGN KEY (STEP_EXECUTION_ID)
		REFERENCES BATCH_STEP_EXECUTION(STEP_EXECUTION_ID)
);

-- 💡 Step이 실패했을 때 어디까지 처리했는지 저장 (재시작 지점)

-- ============================================================================
-- 6. BATCH_JOB_EXECUTION_CONTEXT: Job ExecutionContext
-- ============================================================================
CREATE TABLE BATCH_JOB_EXECUTION_CONTEXT  (
	JOB_EXECUTION_ID BIGINT NOT NULL PRIMARY KEY,
	SHORT_CONTEXT VARCHAR(2500) NOT NULL,
	SERIALIZED_CONTEXT LONGVARCHAR,           -- JSON 직렬화된 데이터
	CONSTRAINT JOB_EXEC_CTX_FK FOREIGN KEY (JOB_EXECUTION_ID)
		REFERENCES BATCH_JOB_EXECUTION(JOB_EXECUTION_ID)
);

-- 💡 모든 Step이 공유하는 데이터 저장

-- ============================================================================
-- 시퀀스 (ID 자동 증가용)
-- ============================================================================
CREATE SEQUENCE BATCH_STEP_EXECUTION_SEQ;
CREATE SEQUENCE BATCH_JOB_EXECUTION_SEQ;
CREATE SEQUENCE BATCH_JOB_SEQ;
```

### 실전 SQL 스크립트

**전체 확인 스크립트** (`docs/batch/sql/step01-verify-tables.sql`):

```sql
-- 1. 테이블 개수 확인
SELECT COUNT(*) AS table_count
FROM INFORMATION_SCHEMA.TABLES
WHERE TABLE_SCHEMA = 'PUBLIC'
  AND TABLE_NAME LIKE 'BATCH%';
-- 예상: 6

-- 2. 테이블 목록 조회
SELECT TABLE_NAME
FROM INFORMATION_SCHEMA.TABLES
WHERE TABLE_SCHEMA = 'PUBLIC'
  AND TABLE_NAME LIKE 'BATCH%'
ORDER BY TABLE_NAME;

-- 3. 각 테이블의 컬럼 확인
SELECT TABLE_NAME, COLUMN_NAME, TYPE_NAME, IS_NULLABLE
FROM INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_NAME LIKE 'BATCH%'
ORDER BY TABLE_NAME, ORDINAL_POSITION;

-- 4. 외래 키 관계 확인
SELECT
    FK.FK_NAME AS foreign_key,
    PT.TABLE_NAME AS parent_table,
    CT.TABLE_NAME AS child_table
FROM INFORMATION_SCHEMA.CROSS_REFERENCES FK
JOIN INFORMATION_SCHEMA.TABLES PT ON FK.PKTABLE_NAME = PT.TABLE_NAME
JOIN INFORMATION_SCHEMA.TABLES CT ON FK.FKTABLE_NAME = CT.TABLE_NAME
WHERE PT.TABLE_NAME LIKE 'BATCH%'
ORDER BY PT.TABLE_NAME;

-- 5. 시퀀스 확인
SELECT SEQUENCE_NAME, CURRENT_VALUE
FROM INFORMATION_SCHEMA.SEQUENCES
WHERE SEQUENCE_NAME LIKE 'BATCH%';
-- 예상: 3개 시퀀스
```

## 💾 3. 테이블의 영속성 (메모리 DB의 한계)

### H2 메모리 DB의 생명주기

```
┌─────────────────────────────────────────────────────────────┐
│            H2 메모리 DB 생명주기                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  [1] 애플리케이션 시작                                        │
│      └─ H2 메모리 DB 생성 (jdbc:h2:mem:batchdb)             │
│          └─ BATCH_* 테이블 6개 생성                          │
│                                                              │
│  [2] 애플리케이션 실행 중                                     │
│      └─ 테이블과 데이터가 메모리에 존재                       │
│          └─ H2 콘솔 접속 가능 ✅                             │
│                                                              │
│  [3] 애플리케이션 종료                                        │
│      └─ H2 메모리 DB 소멸 💥                                 │
│          └─ 모든 테이블과 데이터 삭제                         │
│                                                              │
│  [4] 애플리케이션 재시작                                      │
│      └─ 새로운 H2 메모리 DB 생성 (깨끗한 상태)               │
│          └─ BATCH_* 테이블 다시 생성                         │
│              └─ 이전 데이터는 없음!                          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 실험으로 확인하기

#### 실험 1: 애플리케이션 종료 후 H2 콘솔 접속 시도

```bash
# 터미널 1: 애플리케이션 실행
cd spring-batch-onboarding
../gradlew bootRun

# 웹 브라우저: H2 콘솔 접속 성공 ✅
http://localhost:8080/h2-console

# 터미널 1: Ctrl+C로 애플리케이션 종료

# 웹 브라우저: H2 콘솔 새로고침
# → "Database not found" 에러 발생! ❌
```

**왜?** 메모리 DB는 프로세스와 생명주기를 같이하기 때문!

#### 실험 2: 데이터 영속성 테스트

**시나리오:**
1. 애플리케이션 시작 → Job 실행 → BATCH_JOB_INSTANCE에 데이터 1건 삽입
2. 애플리케이션 종료
3. 애플리케이션 재시작
4. BATCH_JOB_INSTANCE 조회

**예상 결과:**
```sql
SELECT COUNT(*) FROM BATCH_JOB_INSTANCE;
-- 결과: 0 (이전 데이터는 사라짐)
```

**이유:** 메모리 DB는 **휘발성(Volatile)** 저장소입니다.

### 메모리 DB vs 파일 DB vs RDBMS

| 항목 | H2 메모리 | H2 파일 | MySQL/PostgreSQL |
|------|----------|---------|------------------|
| **URL 형식** | `jdbc:h2:mem:batchdb` | `jdbc:h2:file:./data/batchdb` | `jdbc:mysql://localhost:3306/batchdb` |
| **데이터 위치** | JVM 힙 메모리 | 로컬 파일 시스템 | 별도 DB 서버 |
| **영속성** | ❌ 앱 종료 시 소멸 | ✅ 파일로 저장 | ✅ 서버에 저장 |
| **속도** | 🚀 매우 빠름 | ⚡ 빠름 | ⚙️ 보통 |
| **설치** | 불필요 | 불필요 | 별도 설치 필요 |
| **용도** | 개발, 테스트 | 로컬 개발 | 운영 환경 |

### DB_CLOSE_DELAY=-1의 의미

```yaml
spring:
  datasource:
    url: jdbc:h2:mem:batchdb;MODE=MySQL;DB_CLOSE_DELAY=-1
```

**DB_CLOSE_DELAY 옵션:**

| 값 | 의미 | 동작 |
|----|------|------|
| `-1` | JVM 종료 전까지 유지 | 마지막 연결이 끊겨도 DB 유지 |
| `0` (기본값) | 마지막 연결 종료 시 DB 삭제 | H2 콘솔 연결 끊으면 즉시 소멸 |
| `10` | 10초 후 DB 삭제 | 마지막 연결 종료 10초 후 소멸 |

**실무 시나리오:**

```
DB_CLOSE_DELAY=0인 경우:

1. 애플리케이션 시작 (DB 생성)
2. H2 콘솔 접속 (연결 1개)
3. H2 콘솔 로그아웃 (연결 0개)
4. DB 즉시 소멸! 💥
5. 애플리케이션이 Job 실행 시도
6. "Table BATCH_JOB_INSTANCE not found" 에러!
```

**-1로 설정하면:**
- H2 콘솔을 닫아도 DB 유지
- 애플리케이션이 종료될 때만 DB 소멸

### 데이터를 영속적으로 보관하려면?

#### 방법 1: H2 파일 모드 사용

```yaml
spring:
  datasource:
    url: jdbc:h2:file:./data/batchdb  # 파일로 저장
```

**장점:**
- 애플리케이션 재시작해도 데이터 유지
- 별도 DB 서버 불필요

**단점:**
- 파일 손상 가능성
- 운영 환경 부적합

#### 방법 2: MySQL/PostgreSQL 사용 (권장)

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/batchdb
    username: root
    password: password
    driver-class-name: com.mysql.cj.jdbc.Driver

  batch:
    jdbc:
      initialize-schema: never  # 테이블 수동 관리
```

## 🎯 Key Takeaways

### 1. 테이블 생성은 Spring Batch가 자동으로 처리
- `@EnableBatchProcessing` + `initialize-schema: always`
- jar 내부의 DB별 DDL 스크립트 실행

### 2. H2 메모리 DB는 휘발성
- 애플리케이션 종료 시 모든 데이터 소멸
- `DB_CLOSE_DELAY=-1` 설정으로 조기 소멸 방지

### 3. H2 콘솔은 애플리케이션 실행 중에만 접속 가능
- 메모리 DB는 프로세스와 생명주기를 같이함
- 접속 정보: `jdbc:h2:mem:batchdb` / `sa` / (비밀번호 없음)

### 4. 운영 환경에서는 반드시 영속성 DB 사용
- H2는 개발/테스트 전용
- MySQL, PostgreSQL 등 RDBMS 사용
- `initialize-schema: never` + DBA가 수동으로 스키마 관리

### 5. 메타데이터 테이블은 Spring Batch의 핵심
- 실행 이력 관리
- 재시작 지원
- 모니터링 및 운영

## ❓ 자주 묻는 질문 (FAQ)

### Q1: H2 콘솔 접속 시 "Database not found" 에러

**A:** 애플리케이션이 실행 중인지 확인하세요. 메모리 DB는 앱이 실행 중일 때만 존재합니다.

### Q2: 테이블이 7개가 아니라 6개만 보여요

**A:** 정상입니다. 6개가 맞습니다. 시퀀스(SEQUENCE)는 테이블이 아닙니다.

### Q3: MySQL로 변경하면 어떻게 되나요?

**A:** DDL이 자동으로 `schema-mysql.sql`로 전환됩니다. 테이블 구조는 동일하지만 문법이 MySQL에 맞게 조정됩니다.

### Q4: 운영 환경에서도 initialize-schema: always를 쓰나요?

**A:** 절대 안 됩니다! 운영에서는 `never`로 설정하고, DBA가 DDL을 수동으로 실행합니다.

### Q5: 데이터를 파일로 저장하려면?

**A:** `jdbc:h2:file:./data/batchdb`로 변경하면 파일 모드로 전환됩니다.
