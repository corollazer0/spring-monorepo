# [Batch Step 13 캡스톤] 일일 정산 Job — 요구사항 명세 & 평가 체크리스트

> 프로덕션 코드는 완성되어 제공됩니다 (`job/settlement/`, `processor/SettlementProcessor`,
> `domain/{DailyTxSummary, Settlement}`) — **여러분의 과제는 테스트 전략 수립과 작성**입니다.

---

## 1. 기능 요구사항

### R1. 일일 정산 — `dailySettlementJob` (파라미터: targetDate)
- 해당 날짜(`transaction_date = targetDate`)의 거래를 **회원별로 집계**해 settlement에 적재한다
- 집계 항목: 총입금(total_deposit), 총출금(total_withdraw), 순액(net_amount = 입금-출금, **음수 가능**), 거래건수(transaction_count)
- 거래가 없는 날짜는 0건 적재로 **정상 종료**한다 (실패 아님)

### R2. 재실행 안전 (보정 시나리오)
- 거래 데이터 보정 후 같은 날짜를 다시 정산할 수 있어야 한다
- 재정산 시 **중복 적재가 없어야** 한다 — clearStep(해당 날짜 삭제)이 선행되는 구조

### R3. 정렬/결정성
- 집계는 member_id 순으로 처리된다 (ORDER BY)

## 4. 평가 체크리스트 (자가 채점)

### 전략 설계 (가장 중요!)
- [ ] 각 검증이 "왜 그 방식인지" 설명할 수 있다
- [ ] 계산 케이스는 단위에, 통합은 조립·멱등성에 — 중복 없이 분담했다

### 계산 규칙 (단위)
- [ ] 순액 계산 + **모든 필드 이관** 검증 (바뀐 것 + 보존된 것)
- [ ] 음수 순액(m3)과 0 순액(m4) 경계를 다뤘다
- [ ] 주입된 settlementDate가 결과에 박히는지 확인했다

### 집계 리더 (단독 또는 통합)
- [ ] 대상 날짜의 거래"만" 집계된다 (다른 날짜 6건 미포함)
- [ ] 회원별 GROUP BY 결과 5행, member_id 오름차순

### Job 통합
- [ ] removeJobExecutions + **settlement 원상복구**(@BeforeEach)를 갖췄다
- [ ] 적재 5건 + 대표 행의 "실제 INSERT 값"(음수 포함) 검증
- [ ] **보정 재실행 → 여전히 5건** (clearStep의 존재 이유 증명)
- [ ] 거래 없는 날짜 → 0건 + COMPLETED
- [ ] 재실행용 run 파라미터로 JobInstance를 분리했다 (Step 3 함정 회피)

### 공통 품질
- [ ] 테스트명 `{대상}_{시나리오}_{예상결과}`, AAA 주석, AssertJ
- [ ] 프로덕션을 일부러 망가뜨렸을 때(예: 순액 계산을 +로) 테스트가 깨진다 (변이 검증)
- [ ] 전체 스위트 두 번 연속 실행해도 통과한다
