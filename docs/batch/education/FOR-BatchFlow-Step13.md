## 1. Before We Start — 정답지 없이

Step 1~12에서 배운 것: 장부와 인프라(1), Job 구조(2~5), Chunk와 읽기/가공/쓰기(6~10),

> **"일일 정산 배치 만들었는데요, 테스트는 뭘 어떻게 짜죠?"**

50-Step 커리큘럼의 최종 프로젝트(Step 43~46)를 압축한 이 캡스톤에서,
프로덕션 코드와 요구사항만 주어집니다. 전략 수립부터 작성까지 — 여러분의 설계입니다.

## 2. What We're Building

```
dailySettlementJob (targetDate)
  clearStep      : 해당 날짜 정산 삭제 — 보정 재실행의 멱등성 (설계된 멱등성!)
  settlementStep : GROUP BY 집계 리더 → 순액 계산 프로세서 → INSERT 라이터
```

## 3. Core Concepts — 전략 수립의 사고 과정 (시연 한 토막)

**예: "재실행해도 정산이 중복되지 않는다"는 어디서 검증하나?**

1. 이 보장의 주체는? → `clearStep`(같은 날짜 삭제 후 재적재)이라는 **구조**
2. 구조의 검증은 단위로 가능한가? → 아니다 — Step 두 개의 **협동**이므로 통합
3. 통합 시나리오는? → 1차 정산(5건) → 같은 날짜 재정산(run 파라미터만 변경) → **여전히 5건**
4. 계산 규칙(음수/0 순액)도 통합에서? → 아니다 — 그건 Processor 순수 단위의 몫

이 사고를 요구사항 전부에 적용하는 것이 과제입니다. 참고로 이 Job의 멱등성은
Step 10(상태 전이 WHERE)과 **다른 방식** — "지우고 다시 만드는" 명시적 멱등성입니다.
두 방식을 비교해보는 것도 좋은 학습입니다.

## 5. Testing — 최종 검증 두 가지 (TestCraft 캡스톤과 동일)

1. **변이 검증**: `SettlementProcessor`의 순액 계산을 `+`로 바꾸면 어떤 테스트가 깨지나?

## 6. Lessons Learned — 캡스톤에서 빠지기 쉬운 함정

- **모든 것을 통합으로**: 음수/0 순액 케이스까지 Job을 돌려 검증 → 느리고 장황.
  계산은 단위로 내려라.
- **카운트만 보고 값을 안 봄**: "5건 적재"는 절반의 검증 — 음수 순액이 DB에
  **그대로** 박혔는지(절대값 보정 사고!)까지 봐야 한다.
- **재실행 테스트에서 Step 3 함정**: 같은 파라미터 재실행은 거부된다 —
  보정 재실행은 run 파라미터로 새 인스턴스를 만들어야 한다.

### 시니어의 시선

> 정산 배치는 회사에서 가장 보수적으로 다뤄야 하는 코드입니다 — 돈이니까요.
> 그래서 정산 테스트의 황금률: **기대값을 코드로 계산하지 말고 손으로 계산해 박아라.**
> 30000-5000=25000을 테스트 안에서 또 빼기로 계산하면, 프로덕션과 같은 버그를
> 테스트도 가질 수 있습니다. 시드 기준값 표가 "손 계산"인 이유입니다.

## 7. Key Takeaways

- 테스트 전략 = 검증 × 방식 매트릭스: 계산은 단위, 집계는 리더, 구조(멱등)는 통합
- 멱등성의 두 방식: 상태 전이 WHERE(Step 10) vs 지우고 다시 만들기(clearStep)
- 정산 검증은 값까지 — 음수가 음수로 남아있는가
- 기대값은 손으로 계산해 박아라 (테스트가 프로덕션 버그를 복제하지 않게)
