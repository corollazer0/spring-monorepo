# [심화 Step 18 캡스톤] 대량 알림 발송 Job — 요구사항 명세 & 평가 체크리스트

> 프로덕션 코드는 완성되어 제공됩니다 (`job/massnotify/` 3개 파일) —
> **여러분의 과제는 테스트 전략 수립과 작성**입니다 (Step 13과 같은 형식).
> 단, 이번엔 심화 트랙(14~17)의 무기가 검증 대상입니다.

## 1. 기능 요구사항

### R1. 대상자 — `massNotificationJob` (파라미터: campaign)
- **ACTIVE 회원 전원(30명)** 에게 캠페인 알림을 발송하고 notification_history에 적재한다
- DORMANT/WITHDRAWN 회원에게 발송되면 안 된다 — **마케팅 발송 사고**

### R2. 개인화 메시지
- 형식: `{campaign}: {이름}님, 고객님만을 위한 혜택이 도착했습니다`
- 캠페인 접두가 곧 멱등 키다 (R4)

### R3. 분할 병렬 처리
- ID 범위 3분할(Partitioning) — 워커마다 자기 범위만 (Step 15의 파티셔너 재사용)

### R4. 재발송 방지 (자연 멱등)
- 같은 캠페인을 다시 실행해도 **이미 받은 회원에게 또 가면 안 된다** (0건 읽고 정상 종료)
- 캠페인이 다르면 별도 발송이다 (멱등 키 = 캠페인)

### R5. 부분 실패 허용 (skip)
- 특정 회원의 발송 실패(수신 불가 주소 등)가 **나머지의 발송을 막으면 안 된다**
- 단, 실패가 한도(5건)를 넘으면 개별 문제가 아니라 시스템 문제 — **Job이 실패해야 한다**
- 실패자는 적재되지 않는다 → 원인 해소 후 같은 캠페인 재실행 시 **실패자만** 발송된다 (R4의 보상)

## 4. 평가 체크리스트 (자가 채점)

### 전략 설계 (가장 중요!)
- [ ] R1~R5 각각이 어느 테스트에서 봉인되는지 1:1로 짚을 수 있다
- [ ] "왜 Async(16)가 아니라 Partitioning(15)인가"를 설명할 수 있다 (해설 4-1)

### 전량 발송 (R1~R3)
- [ ] COMPLETED + StepExecution 구조 (manager 1 + worker 3)
- [ ] 워커 read 합계 = 30 — **manager를 합산에 넣지 않았다** (manager는 워커 집계를 들고 있다!)
- [ ] DB 30건 + 수신자 전원 ACTIVE (JOIN 교차 — 발송 사고 봉인)
- [ ] 대표 1건의 실제 메시지 검증 (접두 + 이름 — 개인화의 증거)

### 멱등 (R4)
- [ ] 같은 캠페인 재실행 → 두 번째는 read 0 + 여전히 30건 ("멱등은 조용하다")
- [ ] 다른 캠페인 → 60건 (멱등 키의 경계 확인)
- [ ] 재실행에 유니크 run 파라미터를 썼다 (Step 3 함정 회피)

### 부분 실패 (R5)
- [ ] 2명 실패 → COMPLETED + 28건 + 워커 skip 합계 2 + 실패자 미적재
- [ ] **복구 시나리오**: 스위치 해제 후 재실행 → "그 2명만" 읽혀 30건 완성
- [ ] 한도 초과(6 > 5) → FAILED (조용한 대량 누락 방지)

### 공통 품질
- [ ] @BeforeEach: removeJobExecutions + notification_history 정리 + **FAIL_MEMBER_IDS.clear()**
- [ ] @AfterEach에도 교보재 정리 (실패한 테스트가 다음 테스트를 오염시키지 않게)
- [ ] ⚠️ JobExecution을 반환하는 헬퍼 메서드를 만들지 않았다
      (@SpringBatchTest의 리스너가 잡 스코프 팩토리로 오인 — 진단표 참조)
