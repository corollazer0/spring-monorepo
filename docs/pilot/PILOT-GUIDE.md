# 학습자 파일럿 운영 가이드 (운영자용)

> 세 학습 모듈(TestCraft/BatchFlow/WebFlow)을 실제 학습자에게 처음 적용할 때의
> 준비·운영·피드백 수집 절차. 파일럿의 목적은 "가르치기"가 아니라 **커리큘럼 검증**이다 —
> 학습자가 막히는 곳이 곧 커리큘럼의 버그다.

---

## 0. 1회차 시작 시퀀스 (준비물은 전부 갖춰져 있다)

```
① 대상자 1~2명 선정 + 일정 확정 (2장 페르소나 기준)            ← 운영자만 할 수 있는 일
② Day 0 사전 점검 1회 (3장 체크리스트 — 같은 망에서 클론→테스트→bootRun)
③ 학습자에게 LEARNER-KIT.md 전달 + 첫 미팅에서 규칙 4가지 구두 확인
④ 학습자가 포크에 MY-PILOT-LOG.md 생성 (PILOT-LOG-TEMPLATE.md 복사)
⑤ 주 2회 체크인 시작 (6장 루틴)
```

| 준비물 | 파일 |
|--------|------|
| 학습자 전달 키트 (1페이지) | [LEARNER-KIT.md](./LEARNER-KIT.md) |
| 진행 로그 템플릿 (학습자 기입) | [PILOT-LOG-TEMPLATE.md](./PILOT-LOG-TEMPLATE.md) |

## 1. 목적과 성공 기준

**검증 대상** (우선순위순):
1. exercise 난이도 — TODO 지시문만으로 풀 수 있는가, answer 없이는 불가능한가
2. 문서 분량/이해도 — FOR 문서를 읽고 example을 이해할 수 있는가
3. 표기 소요시간의 정확도 — "1.5h"가 실제로 1.5h인가
4. 환경 진입 장벽 — README만으로 30분 안에 첫 테스트가 도는가

**정량 지표** (파일럿 종료 시 산출):

| 지표 | 목표 | 측정 방법 |
|------|------|----------|
| 완주율 (필수 코스) | 80%↑ | Step 완료 보고 |
| Step 소요시간 실측/표기 비율 | 0.7~1.5배 | Step별 미니 설문 |
| 막힌 지점 중 자가 해결율 (문서/진단표로) | 70%↑ | 막힘 기록 |
| 환경 셋업 소요 | 30분 이내 | Day 0 기록 |

## 2. 대상자 선정과 범위

- **인원**: 1~2명 (피드백 품질 > 수량. 3명 이상은 2회차에)
- **페르소나 매칭** (모듈 대상과 일치해야 검증이 유효):
  - TestCraft: 테스트 코드를 써본 적 없는 1~3년차 Spring 개발자
  - BatchFlow/WebFlow: TestCraft 수료자 (파일럿 1회차에선 보통 도달 못 함)
- **1회차 범위 권장**: **TestCraft 필수 코스(Step 1~9, 12)만** — 약 2주(업무 병행 1~2h/일).
  심화(10~14)와 Batch/Web은 완주자가 원할 때 연장. 범위를 좁혀야 피드백이 깊어진다.
- **AI 협업 검증 포함 여부 결정**: 학습자가 Claude Code를 쓴다면 루트 CLAUDE.md의
  "AI 요청 가이드"와 에이전트/스킬 체계도 검증 대상이 된다 — 포함 시 별도 관찰 항목 추가
  (예: AI에게 모듈을 올바르게 명시하는가, answer를 AI로 생성해버리지 않는가 — 학습 규칙 안내 필요)

## 3. 환경 준비 체크리스트 (Day 0, 운영자가 사전 확인)

- [ ] 레포 접근 권한 — GitHub 계정 초대 (private인 경우)
- [ ] 학습자 PC: JDK 8+ (11 권장), Git, IntelliJ Community — 모듈 README 0장 기준
- [ ] **사내망 변수 사전 점검**: Gradle 의존성 다운로드(프록시/사설 인증서), github.com 접근
  — 막히면 첫날이 통째로 증발한다. 가능하면 운영자가 같은 망에서 클론→빌드 1회 선행
- [ ] 클론 후 `.\gradlew :spring-test-onboarding:test` 그린 확인 (첫 빌드는 수 분 소요 안내)
- [ ] IntelliJ 인코딩 UTF-8 설정 안내 (한글 주석 — README 3장)
- [ ] Windows 학습자: PowerShell에서 `.\gradlew` 표기 안내

## 4. 학습자 작업 방식 (중요한 사전 결정)

**개인 포크(권장) 또는 개인 브랜치에서 작업한다.**
exercise는 `@Disabled`를 제거하고 푸는 구조라서, 푼 결과가 main에 머지되면
**다음 학습자가 풀 문제가 사라진다.** main은 항상 "미개봉" 상태를 유지할 것.

- 포크 방식: 학습자가 포크 → 자유롭게 커밋 → 운영자는 포크를 보고 진도/코드 확인
- 커밋 권장: Step 완료마다 1커밋 (레포의 커밋 규약을 따라 해보는 것 자체가 연습)

## 5. 학습자에게 전달할 시작 키트

```
① spring-test-onboarding/README.md       ← 여기서 시작 (환경 + 30초 검증 + 웹앱 실행)
② docs/test/curriculum/00-TestCraft-Curriculum.md  ← 전체 지도 + 진행 루틴
③ docs/test/education/FOR-Test-Step01.md  ← 첫 Step
```

**학습 규칙 4가지** — [LEARNER-KIT.md](./LEARNER-KIT.md) 3장에 명문화되어 있다
(포크 작업 / 30분 룰 / 막힘 즉시 기록 / AI는 설명만·답 생성 금지).
첫 미팅에서 키트를 같이 열고 구두로 한 번 확인할 것.

## 6. 운영 루틴과 피드백 수집

**운영 주기**: 주 2회 15분 체크인 (비동기 채널 + 주 1회 대면/콜 권장)

**Step별 미니 설문** (Step 완료 보고 시, 1분 분량):

```
Step N 완료 — 소요시간: __h / 난이도(1쉬움~5어려움): _ /
막힌 곳: (없음 | 있었음 → 어디서, 무엇으로 해결: 문서/진단표/answer/검색/질문) /
문서에서 부족했던 한 가지: ___
```

**막힘 기록 분류** (운영자가 집계): ① 환경 문제 ② 문서 불충분 ③ exercise 지시문 모호
④ 선수 지식 부족 ⑤ 레포 버그 — ②③⑤가 커리큘럼 수정 대상, ①은 README FAQ 후보,
④는 선수 과정 안내 보강 후보

**완주 후 종합 인터뷰** (30분):
- 가장 가치 있었던 Step / 없어도 됐을 Step과 이유
- "일주일" 약속은 지켜졌나 — 어디서 밀렸나
- 완주 체크리스트의 자가 점검 질문에 실제로 답할 수 있는가 (구두 확인 2~3개 샘플)
- 동료에게 권하겠는가 (1~10) + 한 가지를 고친다면

## 7. 피드백 반영 절차 (기존 운영 규칙에 연결)

1. 파일럿 종료 → 피드백 정리를 `docs/test/plan/plan.md`에 새 Plan으로 기록 (의사결정 포함)
2. 수정 작업은 기존 규칙대로: plan/task 등록 → Step 단위 수정 → 매 커밋 그린
3. 환경/도구 함정은 `spring-troubleshoot` 진단표와 모듈 README FAQ에 자산화
4. 난이도 조정 원칙: exercise가 너무 어려우면 **TODO 지시문을 쪼개고 힌트를 추가**
   (문제 자체를 쉽게 만들기 전에) / 너무 쉬우면 보너스 과제를 FOR 문서에 추가

## 8. 리스크와 대비

| 리스크 | 대비 |
|--------|------|
| 사내망에서 Gradle 의존성 차단 | Day 0 사전 점검 (3장) — 최악 시 핫스팟/예외 신청 경로 확보 |
| 업무 병행으로 진도 정체 | 주 2회 체크인에서 조기 감지 — "중단"도 유효한 데이터 (어느 Step에서?) |
| answer 복붙으로 형식적 완주 | 30분 룰 + 체크인에서 샘플 질문 ("이 never()는 왜 있나요?") |
| 학습자가 main에 직접 커밋 | 포크 방식 강제 (4장) — 권한 자체를 read-only로 |
| AI로 exercise를 통째로 풀어버림 | 학습 규칙으로 명시: AI는 "설명 요청"에만, answer 생성 금지 (2장 결정 사항) |

---

> **파일럿 운영자의 한 줄 원칙**: 학습자를 평가하지 말고 커리큘럼을 평가하라.
> 학습자가 막힌 모든 지점은 학습자의 실패가 아니라 문서·exercise의 개선 기회다.
