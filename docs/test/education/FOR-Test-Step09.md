## 3. Core Concepts — 전략 수립의 사고 과정

요구사항 하나를 골라 "어디서 검증할까"를 정하는 사고 흐름을 시연합니다.

**예: "존재하지 않는 게시글에는 댓글을 달 수 없다 (404)"**

1. 이 규칙의 **판단 주체**는 누구인가? → `CommentService.createComment`의 if문
2. 그럼 판단 로직 자체는? → **Mockito 단위**: boardDao Mock이 null을 주면 예외 + insert 미호출
3. 그 예외가 404로 **번역**되는 배선은? → **@WebMvcTest**: Mock Service가 던지면 404 + message
4. E2E에서도 또 검증해야 하나? → 아니다. 연결은 위의 둘로 충분하다. E2E는 정상 여정에 집중.

이렇게 **하나의 요구사항이 계층별로 "조각나서" 검증**됩니다. 각 계층은 자기 관심사만:

| 계층 | 질문 |
|------|------|
| 단위 | 판단이 맞는가 |
| DAO | SQL이 맞는가 |
| Controller | HTTP 계약과 배선이 맞는가 |
| E2E | 여정이 연결되는가 |

워밍업 테스트의 5가지 질문에 먼저 답해보세요 — 답이 술술 나오면 준비된 겁니다.

## 5. Testing — 최종 검증 두 가지

**검증 1 — 변이 테스트(수동)**: 프로덕션 코드를 일부러 망가뜨려 보세요.
- `CommentService`의 소유자 검증 if문을 주석 처리 → 여러분의 어떤 테스트가 깨지나요?
- `CommentMapper.xml`의 `ORDER BY comment_id ASC`를 DESC로 → 깨지는 테스트가 있나요?
- **깨지는 테스트가 없다면 그 규칙은 보호받지 못하는 겁니다.** 테스트를 보강하세요.

**검증 2 — 반복 실행**: `:spring-test-onboarding:test`를 **두 번 연속** 돌리세요.
두 번째가 실패하면 E2E 정리가 빠진 겁니다 (Step 8의 교훈).

## 6. Lessons Learned — 캡스톤에서 흔히 빠지는 함정

### 함정 1: 모든 것을 E2E로
"E2E가 제일 확실하니까"라며 요구사항 전부를 E2E로 작성 → 느리고, 실패 시 원인을 못 찾고,
경계 케이스(빈 내용, 타인 삭제...)마다 로그인 절차가 반복됩니다.
**확실함의 대가는 피드백 속도** — 피라미드를 기억하세요.

### 함정 2: 프로덕션 코드를 안 읽고 테스트부터
Service의 if문(= 비즈니스 규칙)을 안 읽으면 무엇을 검증할지 알 수 없습니다.
**테스트 케이스의 절반은 프로덕션 코드의 분기에서 나옵니다.** 나머지 절반은 요구사항 문서에서.

### 함정 3: 통과하는 테스트에 만족
체크리스트 마지막 항목이 괜히 있는 게 아닙니다 — **일부러 망가뜨려 깨지는지** 확인하기 전까지
그 테스트는 검증되지 않은 코드입니다. (Step 1의 교훈이 마지막까지 따라옵니다)

### 시니어의 시선

> 캡스톤을 마친 여러분에게 마지막 조언: 실무에서는 "테스트 짤 시간이 없다"는 말을
> 수없이 듣게 될 겁니다. 그때 기억하세요 — 테스트는 시간을 쓰는 게 아니라
> **디버깅과 수동 확인에 쓸 시간을 앞당겨 지불**하는 겁니다. 그리고 이 모듈에서
> 경험했듯, 익숙해지면 테스트 작성은 빠릅니다. 느린 건 처음뿐입니다.

---

## 7. Key Takeaways

- 테스트 전략 = 요구사항 × 계층 매트릭스: 판단은 단위, SQL은 DAO, 계약은 Controller, 연결은 E2E
- 프로덕션 코드의 분기(if문)가 테스트 케이스의 절반이다
- 변이 테스트(일부러 망가뜨리기)로 테스트의 보호력을 검증하라
- 두 번 연속 실행되지 않는 테스트는 완성이 아니다
